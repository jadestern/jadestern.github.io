옥슬린트 정식 출시

이제 옥슬린트가 정식 버전으로 출시되었다는 기쁜 소식을 알려드립니다! 이 이정표는 문제를 신속하게 해결하고 분류할 수 있는 저희 팀의 능력을 의미합니다.

옥슬린트는 기본적으로 설정할 필요 없이 오류나 쓸모없는 코드를 포착하도록 설계된 JavaScript 린터입니다.

사용 방법

현재로서는 옥슬린트가 ESLint를 완전히 대체할 수 있는 것은 아니며, ESLint의 느린 속도가 워크플로우에 병목 현상을 일으킬 때 이를 개선하는 역할을 합니다.

더 빠른 피드백 루프를 원한다면 대규모 코드베이스에서 실행하는 데 몇 초밖에 걸리지 않으므로 린트 단계 또는 CI 설정에서 ESLint보다 먼저 oxlint를 실행하는 것이 좋습니다.

JavaScript/TypeScript 코드베이스에서 oxlint를 테스트하려면 리포지토리의 루트 디렉터리에서 다음 명령을 실행하기만 하면 됩니다:

sh

$ npx oxlint@latest
$ npx oxlint@latest

sh

$ pnpm dlx oxlint@latest
$ pnpm dlx oxlint@latest

sh

$ yarn dlx oxlint@latest
$ yarn dlx oxlint@latest

sh

$ bunx oxlint@latest
$ bunx oxlint@latest

sh

$ deno run npm:oxlint@latest
$ deno run npm:oxlint@latest

또는 설치 가이드에서 자세한 지침을 참조하세요.

디자인

ESLint보다 50~100배 빠른 속도

실제 시나리오에서 Shopify는 75분 소요되던 ESLint 실행 시간이 이제 10초밖에 걸리지 않는다고 보고했습니다.

제이슨 밀러, Shopify DX 및 Preact 제작자:

옥슬린트는 Shopify에 큰 도움이 되었습니다. 이전 린팅 설정은 실행하는 데 75분이 걸렸기 때문에 CI 작업자 40명 이상이 분산해서 사용해야 했습니다.

이에 비해 oxlint는 단일 작업자에서 동일한 코드베이스를 린팅하는 데 약 10초가 소요되며, 결과물을 해석하기가 더 쉽습니다.

심지어 마이그레이션할 때 이전 설정에서 숨겨지거나 건너뛰었던 몇 가지 버그를 발견하기도 했습니다!

성능 향상의 대부분은 Rust와 병렬 처리를 핵심 요소로 활용하여 의도적으로 성능을 위해 설계된 Oxlint에서 비롯된 것입니다.

정확성을 위한 린트

옥슬린트는 기본적으로 오류, 중복 또는 혼란스러운 코드를 식별하여 불필요한 니트 피킹 규칙( perf, suspicious, pedantic 또는 style 으로 분류됨)보다 정확성을 우선시하며, 기본적으로 비활성화되어 있습니다.

사용 편의성

새로운 JavaScript/TypeScript 코드베이스를 설정하는 일은 점점 더 복잡해지고 있습니다. 도구 간에 호환성 문제가 발생할 가능성이 높고, 이로 인해 많은 시간을 낭비할 수 있습니다.

그렇기 때문에 옥슬린트는 기본적으로 설정이 필요 없도록 설계되었으며, Node.js도 필요하지 않습니다. 대부분의 조정은 명령줄을 통해 수행할 수 있으며, ESLint 구성 파일에서 읽는 기능은 현재 작업 중입니다.

향상된 진단

린터 메시지를 이해하는 것은 어려울 수 있습니다. 옥슬린트는 근본 원인을 정확히 찾아내고 유용한 메시지를 제공함으로써 이를 간소화하여 긴 규칙 문서를 읽을 필요가 없도록 하여 귀중한 시간을 절약하는 것을 목표로 합니다.

통합 규칙

옥슬린트는 아직 플러그인 시스템을 제공하지 않지만 TypeScript, React, Jest, Unicorn, JSX-a11y 및 Import와 같은 인기 플러그인의 규칙을 적극적으로 통합하고 있습니다.

저희는 JavaScript 생태계에서 플러그인의 중요성을 인식하고 있으며 DSL 기반 플러그인 시스템도 검토하고 있습니다.

하지만 플러그인 종속성 목록을 관리하거나 호환성 문제를 탐색하거나 버전 제약으로 인해 포크된 플러그인에 의존할 필요가 없는 독립형 린터를 선호하실 수도 있습니다.
