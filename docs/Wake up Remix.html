지난 여름 React Conf에서 우리는 Remix가 "낮잠을 자게 될 것"이라고 발표했습니다.

Remix v2는 React Router를 얇게 감싸고 있었기 때문에 두 프로젝트 사이에 인위적인 분리가 발생했습니다. 우리는 번들러와 서버 런타임 코드를 Remix에서 React Router v7로 직접 옮겨 프로젝트를 간소화하여 Remix를 훌륭하게 만든 모든 것을 React Router v7 "프레임워크 모드"로 병합했습니다. 또한 React Router에서 RSC에 대한 제한적인 지원을 추가할 것이라고 발표했습니다.

그 작업을 완료하여 지난 11월에 v7을 출시하고 지난 주에 RSC 지원을 미리 선보였습니다. 이로써 Remix v1, v2 또는 이전 버전의 React Router를 기반으로 구축한 모든 사용자에게 장기적인 지원과 함께 안정적인 플랫폼을 제공할 수 있게 되었습니다.

이제 리믹스를 깨울 시간입니다!

정말 좋은 리액트 라우터

우리가 Remix를 발전시키기 시작했을 때, 우리는 이 프레임워크가 풀스택 React 아키텍처에 대한 새로운 해석인 RSC 우선의 React 프레임워크가 될 것으로 예상했습니다.

하지만 그 과정에서 예상치 못한 일이 벌어졌습니다: React Router v7이 정말, 정말 좋아졌습니다.

React Router의 RSC 지원 작업 덕분에 이제 로더와 액션에서 직접 서버 컴포넌트를 반환하는 것을 지원하는 원활하고 점진적인 채택 경로를 갖게 되었습니다. 심지어 서버 전용 라우트를 위한 최고 수준의 스토리를 구축했는데, 이는 원래 Remix의 다음 챕터의 일부로 상상했던 것입니다.

모든 경로가 서버 경로인 경우, React Router v7은 이미 Remix가 지향하는 방향과 매우 유사하게 느껴집니다. 차이점은 구조적인 것이 아니라 미묘한 것입니다.

그리고 기술적으로 견고할 뿐만 아니라 실전 테스트를 거쳤습니다. 리액트 라우터는 현재 약 1,100만 개의 GitHub 프로젝트를 포함하여 Shopify, X.com, GitHub, ChatGPT, Linear, T3Chat 등의 수많은 앱에서 사용되고 있습니다. 이제 마이너 릴리스 한 번만 거치면 전체 React 아키텍처를 완벽하게 지원할 수 있습니다. 전담 팀, 장기적인 지원, 개방형 거버넌스 모델을 통해 앞으로도 계속 성장해 나갈 것입니다.

그렇다면 이는 Remix에 어떤 의미가 있을까요?

다음 버전을 자유롭게 구축할 수 있다는 뜻입니다.

새로운 길

지난 몇 년 동안 웹 플랫폼은 괄목할 만한 발전을 거듭해왔으며, 이를 기반으로 구축하는 도구 역시 눈부시게 발전했습니다. 하지만 많은 것을 얻은 만큼 복잡성도 무겁게 느껴질 만큼 커졌습니다. 때때로 최신 웹 개발은 웹을 구축하는 것이 아니라 도구 체인을 탐색하는 것처럼 느껴질 수 있습니다.

과거에는 너무 파괴적이어서 추진하기 어려운 아이디어도 있었습니다. 하지만 Remix에서는 더 이상 특정 패러다임에 얽매이지 않고 자유롭게 탐색할 수 있습니다.

Remix v3는 웹 프레임워크의 개념을 새롭게 재해석한 버전으로, 수십 년간 쌓아온 웹 구축 경험을 바탕으로 새로운 토대를 마련했습니다. 개발자에게 필요한 기능은 포기하지 않으면서도 단순성, 명확성, 성능에 중점을 두었습니다.

최고 수준의 데이터베이스 드라이버부터 내장된 컴포넌트 라이브러리까지, 함께 잘 작동하지만 독립적으로도 사용할 수 있는 모듈식 툴킷입니다(이제 Reach UI도 깨어날 때입니다!).

이것은 단순한 새 버전이 아니라 새로운 방향입니다. 더 빠르고, 더 간단하며, 웹 자체에 더 가까운 방향입니다.

그러기 위해서는 우리가 제어할 수 없는 추상화 계층에 의존하지 않고 전체 스택을 소유해야 합니다. 즉, 중요한 종속성, 심지어 React조차도 없어야 합니다. 우리는 이미 Shopify, Google 및 수많은 다른 기업에서 많이 사용되고 있는 성숙한 가상 DOM 라이브러리인 Preact의 포크에서 시작하고 있습니다.

더 가볍고 빠르며 웹 작동 방식에 더 잘 부합하는 개발 환경을 원했다면 Remix v3가 바로 여러분을 위해 만들어지고 있습니다.

원칙

아직 프리뷰 릴리스가 준비되지는 않았지만 다음과 같은 원칙을 바탕으로 개발을 진행하고 있습니다:

  1. 모델 우선 개발. AI는 사용자 경험과 개발자 워크플로우 모두에서 인간과 컴퓨터의 상호 작용 모델을 근본적으로 변화시킵니다. LLM을 위해 소스 코드, 문서, 툴링 및 추상화를 최적화합니다. 또한 애플리케이션이 단순히 개발 도구가 아닌 제품 자체에서 모델을 사용할 수 있도록 추상화를 개발하세요.
  2. 웹 API를 기반으로 구축하세요. 스택 전체에서 추상화를 공유하면 사람과 기계 모두 컨텍스트 전환의 양을 크게 줄일 수 있습니다. 유일한 풀 스택 에코시스템인 웹 API와 JavaScript를 기반으로 구축하세요.
  3. 뛰어난 런타임. 번들러/컴파일러/타입젠(및 런타임 전 정적 분석)을 위한 설계는 결국 전체 시스템을 오염시키는 잘못된 API 설계로 이어집니다. 모든 패키지는 정적 분석을 예상하지 않고 설계해야 하며 모든 테스트는 번들링 없이 실행되어야 합니다. 브라우저가 관련되어 있으므로 TypeScript 및 JSX와 같은 간단한 변환을 위한 --import 로더는 허용됩니다.
  4. 종속성을 피하세요. 종속성은 다른 사람의 로드맵에 갇히게 합니다. 종속성을 현명하게 선택하고, 완전히 감싸고, 언젠가는 대부분의 종속성을 자체 패키지로 대체할 수 있을 것으로 기대하세요. 목표는 제로입니다.
  5. 수요 구성. 추상화는 단일 목적이어야 하며 대체가 가능해야 합니다. 컴포저블 추상화는 기존 프로그램에서 쉽게 추가하고 제거할 수 있습니다. 모든 패키지는 다른 컨텍스트와 무관하게 유용하고 문서화되어야 합니다. 새로운 기능은 먼저 새로운 패키지로 시도해야 합니다. 불가능하다면 기존 패키지를 분리하여 컴포저블하게 만들 수 있도록 시도하세요. 그러나 거의 항상 양방향으로 함께 변경되는 긴밀하게 결합된 모듈은 동일한 패키지로 이동해야 합니다.
  6. 응집력 있게 배포하세요. 매우 컴포저블한 에코시스템은 학습과 사용이 어렵습니다. 따라서 패키지를 하나의 패키지로 종속성으로 묶어 배포와 문서화를 위해 하나의 도구 상자(리믹스)로 다시 내보냅니다.
