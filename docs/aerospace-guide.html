에어로스페이스 가이드

이 가이드는 전체적으로 위에서 아래로 읽도록 구성되어 있습니다. 뻔한 부분은 건너뛰어도 됩니다.

1. 설치

1.1. 홈브루 설치(기본 설정)

Homebrew는 macOS용 패키지 관리자입니다.

brew install --cask nikitabobko/tap/aerospace

(선택 사항) 홈브루 패키지가 제공하는 완성을 사용하도록 셸을 구성해야 할 수 있습니다: https://docs.brew.sh/Shell-CompletionAeroSpace는 bash, fish 및 zsh 완성을 제공합니다.

brew install --cask nikitabobko/tap/aerospace@0.12.0

1.2. 수동 설치

  1. 릴리스 페이지에서 사용 가능한 최신 zip을 다운로드합니다.

  2. zip 압축 풀기

  3. 압축을 푼 AeroSpace-v$VERSION/AeroSpace.app 을 다음 위치에 넣습니다. /Applications

  4. 압축을 푼 AeroSpace-v$VERSION/bin/aerospace 을 $PATH에 넣습니다(이 단계는 선택 사항입니다. CLI에서 AeroSpace와 상호 작용할 수 있는 경우에만 필요합니다.)

이 메시지가 표시되는 경우

"AeroSpace.app" can't be opened because Apple cannot check it for malicious software.

문제 해결을 위한옵션 1 

xattr -d com.apple.quarantine /Applications/AeroSpace.app

문제 해결을 위한옵션 2

  1. Finder에서 다음 위치로 이동합니다. /Applications/AeroSpace.app

  2. 마우스 오른쪽 클릭

  3. 열기를 클릭합니다(예, 그렇게 바보 같죠).

2. AeroSpace 구성하기

2.1. 사용자 지정 구성 위치

AeroSpace는 두 위치에서 사용자 지정 구성을 찾으려고 시도합니다:

  1. ~/.aerospace.toml

  2. ${XDG_CONFIG_HOME}/aerospace/aerospace.toml (환경 변수 XDG_CONFIG_HOME 가 제공되지 않으면 ~/.config 로 대체됨).

구성이 두 개 이상의 위치에서 발견되면 모호성이 보고됩니다.

2.3. 기본 설정

기본 구성은 설명서의 일부이며 설명과 함께 모든 사소한 구성 키가 포함되어 있습니다. 사소한 구성 옵션은 이 가이드에서 자세히 설명합니다. 사용자 지정 구성이 없는 경우 AeroSpace는 기본 구성을 로드합니다.

사용자 지정 구성에서 키가 생략된 경우 특정 키에 대해 달리 명시되지 않는 한 기본 구성의 값으로 대체됩니다. 즉, 즉:

  • mode.*.binding. 빈 TOML 테이블로 폴백됩니다. 사용자 설정은 키보드 바인딩에 대한 진실의 원천입니다. 구성에 모든 키보드 바인딩과 바인딩 모드를 명시적으로 언급해야 합니다.

  • on-focused-monitor-changed. 빈 TOML 배열로 폴백됩니다.

  • exec TOML 테이블로 폴백합니다. 참조: 실행-* 환경 변수 (너무 지루하고 장황해서 default-config.toml 에서도 언급하고 싶지 않습니다.)

경험 법칙: 모든 "스칼라 같은" 값은 항상 기본 구성으로 폴백됩니다. 모든 "벡터와 같은" 값은 빈 TOML 배열 또는 테이블로 폴백됩니다.

이렇게 하면 기본값을 좋아하는 사소한 구성 키로부터 구성을 깔끔하게 유지할 수 있습니다. 앱 설치에서 기본 구성을 복사하여 사용자 지정 구성을 부트스트랩할 수 있습니다.

2.4. 바인딩 모드

서로 다른 바인딩 모드를 생성하여 여러 바인딩 세트를 만들 수 있습니다. 다른 바인딩 모드로 전환하면 현재 모드의 모든 바인딩이 비활성화되고 새 모드에 지정된 바인딩만 활성화됩니다. AeroSpace가 시작하는 초기 바인딩 모드는 "기본"입니다.

이 기능은i3의 기능과 완전히 동일합니다.

바인딩 모드 작업은 두 부분으로 구성됩니다. 1. 바인딩 모드로 전환할 바인딩을 정의하고 2. 바인딩 모드 자체를 선언하는 것입니다.

[mode.main.binding]            # Declare 'main' binding mode
    alt-r = 'mode resize'      # 1. Define a binding to switch to 'resize' mode

[mode.resize.binding]          # 2. Declare 'resize' binding mode
    minus = 'resize smart -50'
    equal = 'resize smart +50'

2.5. 명령

명령은 AeroSpace를 조작하고 상태를 쿼리하는 데 사용하는 것입니다.

명령을 사용하는 방법에는 두 가지가 있습니다:

  1. 키를 바인딩하여 AeroSpace 명령을 실행합니다. 예시:

    [mode.main.binding]
    # Bind alt-1 key to switch to workspace 1
    alt-1 = 'workspace 1'
    # Or bind a sequence of commands
    alt-shift-1 = ['move-node-to-workspace 1', 'workspace 1']

  2. CLI에서 명령 실행하기. Terminal.app을 열고 입력합니다:

    aerospace workspace 1

사용 가능한 명령 목록은 명령을 참조하십시오.

2.6. 키보드 레이아웃 및 키 매핑

기본적으로 구성의 키 바인딩은 qwerty 레이아웃으로 인식됩니다.

다른 레이아웃, 다른 알파벳을 사용하거나 기존 키의 멋진 별칭을 사용하려는 경우 key-mapping.key-notation-to-key-code 을 사용할 수 있습니다.

# Define my fancy unicorn key notation
[key-mapping.key-notation-to-key-code]
    unicorn = 'u'

[mode.main.binding]
    alt-unicorn = 'workspace wonderland' # (⁀ᗢ⁀)

  • dvorak 및 colemak 사용자를 위해 AeroSpace는 사전 구성된 사전 설정을 제공합니다.

    [key-mapping]
    preset = 'dvorak'  # or 'colemak'

2.7. 실행-* 환경 변수

exec-* 명령 및 콜백의 환경 변수를 구성할 수 있습니다(예: exec-and-forget, 'exec-on-workspace-change' 콜백).

  • exec.inherit-env-vars = true 는 AeroSpace.app 의 환경 변수를 상속할지 여부를 구성합니다. (기본값은 true 입니다.)

  • 다음 구문을 사용하여 환경 변수를 재정의할 수 있습니다:

    [exec.env-vars]
    PATH = '${HOME}/bin:${PATH}'

    환경 변수 대체는 다음과 같은 형태로 지원됩니다. ${ENV_VAR}

  • list-exec-env-vars 명령을 사용하여 환경 변수의 최종 결과를 검사할 수 있습니다.

  • macOS의 GUI 앱에는 기본적으로 PATH (docs.brew.sh)에 홈브루의 접두사가 없습니다. 따라서 구성에서 exec 섹션을 재정의하지 않으면 AeroSpace는 다음 exec 구성으로 폴백합니다:

    [exec]
    inherit-env-vars = true
[exec.env-vars]
    PATH = '/opt/homebrew/bin:/opt/homebrew/sbin:${PATH}'

3. 트리

AeroSpace는 모든 창과 컨테이너를 트리에 저장합니다. AeroSpace 트리 타일링 모델은 i3에서 영감을 얻었습니다.

정의에서 영감을 받았습니다. 리프가 아닌 각 노드를 "컨테이너"라고 합니다.

경고

 i3의 "컨테이너"는 AeroSpace의 "노드"와 동일한 용어입니다.

  • 각 작업 공간에는 자체 단일 루트 노드가 포함됩니다.

  • 각 컨테이너에는 임의의 수의 자식 노드가 포함될 수 있습니다.

  • Windows는 가능한 유일한 리프 노드입니다. Windows에는 자식 노드가 없습니다.

  • 모든 컨테이너에는 두 가지 프로퍼티가 있습니다:

    1. 레이아웃 (사용 가능한 값: tiles, accordion)

    2. 방향(사용 가능한 값: horizontal, vertical)

여기서 "창의 레이아웃"이란 창의 부모 컨테이너의 레이아웃을 의미합니다.

예시를 보면 트리 타일링 모델을 더 쉽게 이해할 수 있습니다.

그림 2. 복잡한 트리 구조

컨테이너를 원하는 만큼 깊게 중첩할 수 있습니다.

트리에서 네 가지 기본 방향(왼쪽, 아래, 위, 오른쪽)으로 탐색할 수 있습니다. 포커스 명령을 사용하여 탐색할 수 있습니다.

트리 구조는 세 가지 명령으로 변경할 수 있습니다:

  1. move

  2. join-with

  3. split (i3와의 호환성을 위한 것입니다. split 보다 join-with 를 선호하세요.)

3.1. 레이아웃

AeroSpace는 총 4가지 레이아웃을 제공합니다:

  • h_tiles 가로 타일(i3에서는 "가로 분할"이라고 함)

  • v_tiles 수직 타일(i3에서는 "수직 분할"이라고 함)

  • h_accordion 수평 아코디언(i3의 "탭 레이아웃"과 유사)

  • v_accordion 수직 아코디언(i3의 "스택형 레이아웃"과 유사)

이전 섹션에서 이미 tiles 레이아웃에 익숙해졌을 것입니다.

아코디언은 창이 서로 위에 배치되는 레이아웃입니다.

  • 가로 아코디언은 왼쪽과 오른쪽에 패딩을 표시하여 해당 방향에 다른 창이 있음을 시각적으로 나타냅니다.

  • 수직 아코디언은 위쪽과 아래쪽 패딩을 표시하여 해당 방향에 다른 창이 있음을 시각적으로 나타냅니다.

그림 3. 가로 아코디언

그림 4. 수직 아코디언

tiles 레이아웃에서와 마찬가지로 포커스 명령을 사용하여 아코디언 레이아웃을 탐색할 수 있습니다.

focus (left|right) 명령을 사용하여 h_accordion 에서 창을 탐색할 수 있습니다. v_accordion 에서는 focus (up|down) 명령을 사용하여 창을 탐색할 수 있습니다.

아코디언 패딩은 accordion-padding 옵션을 통해 구성할 수 있습니다.

3.2. 정규화

기본적으로 AeroSpace는 두 가지 유형의 트리 정규화를 수행합니다:

  1. 자식이 하나만 있는 컨테이너는 "평탄화"됩니다. 루트 컨테이너는 예외이며, 단일 창 자식을 가질 수 있습니다. 다음과 같이 구성합니다. enable-normalization-flatten-containers

  2. 서로 중첩되는 컨테이너는 방향이 반대여야 합니다. 구성 기준 enable-normalization-opposite-orientation-for-nested-containers

예제 1

첫 번째 정규화에 따르면 이러한 레이아웃은 불가능합니다:

h_tiles (root node)
└── v_tiles
    └── window 1

즉시 다음과 같이 변형됩니다.

v_tiles (new root node)
└── window 1

예제 2

두 번째 정규화에 따르면 이러한 레이아웃은 불가능합니다:

h_tiles
├── window 1
└── h_tiles
    ├── window 2
    └── window 3

즉시 다음과 같이 변환됩니다.

h_tiles
├── window 1
└── v_tiles
    ├── window 2
    └── window 3

정규화를 사용하면 화면에 창이 배치되는 방식을 보고 트리 구조를 더 쉽게 이해할 수 있습니다. 이 줄을 설정에 배치하여 정규화를 비활성화할 수 있습니다:

enable-normalization-flatten-containers = false
enable-normalization-opposite-orientation-for-nested-containers = false

정규화에 대해 잘 알고 있는 하드코어 i3 사용자가 아니라면 정규화를 활성화하는 것이 좋습니다.

3.3. 플로팅 윈도우

일반적으로 플로팅 윈도우는 타일링 트리의 일부로 간주되지 않습니다. 하지만 포커스 명령에서는 그렇지 않습니다.

포커스 명령의 관점에서 플로팅 윈도우는 타일링 트리의 일부입니다. 플로팅 윈도우 부모 컨테이너는 플로팅 윈도우의 중앙을 포함하는 가장 작은 타일링 컨테이너로 결정됩니다.

이 기법을 사용하면 부동 창에 초점을 맞추기 위한 추가 바인딩이 필요하지 않습니다.

4. 가상 작업 공간 에뮬레이션

기본 macOS 스페이스에는 많은 문제가 있습니다.

  • 스페이스 전환 애니메이션이 느립니다.

    • 스페이스 전환 애니메이션을 비활성화할 수 없습니다( Reduce motion 설정을 켜면 약간 더 빠르게 만들 수 있지만 최적이 아닙니다).

  • 스페이스 수가 제한되어 있습니다(모니터 한 대에 최대 16개의 스페이스).

  • 스페이스를 생성/삭제/순서 변경할 수 없으며 핫키를 사용하여 스페이스 간에 창을 이동할 수 없습니다(핫키를 사용하여 스페이스 간에 전환만 가능).

  • Apple은 스페이스와 통신할 수 있는 공용 API를 제공하지 않습니다(스페이스 생성/삭제/순서 변경/스페이스 간 전환 및 스페이스 간 창 이동).

스페이스는 다루기가 매우 어렵기 때문에 AeroSpace는 스페이스를 재구현하여 "작업 공간"이라고 부릅니다. 작업 공간이 활성화되어 있지 않으면 모든 창이 화면의 보이는 영역 바깥쪽, 오른쪽 하단 또는 왼쪽 모서리에 배치됩니다. 작업 공간으로 다시 전환하면(예: 작업 공간 명령 또는 cmd + tab) 창이 화면의 보이는 영역으로 다시 배치됩니다.

AeroSpace를 종료하거나 AeroSpace가 충돌을 감지하면 모든 창이 화면의 보이는 영역으로 다시 배치됩니다.

AeroSpace는 트레이 아이콘(오른쪽 상단 모서리)에 현재 활성화된 작업 공간의 이름을 표시하여 사용자에게 현재 어떤 작업 공간이 활성화되어 있는지 시각적으로 피드백을 제공합니다.

AeroSpace 작업 공간 사용 시 의도된 작업 흐름은 하나의 macOS 공간(또는 Displays have separate Spaces 이 활성화된 경우 보유한 모니터 수만큼)만 사용하고 더 이상 macOS 공간과 상호 작용하지 않는 것입니다.

참고

좋든 나쁘든, macOS는 창을 보이는 영역 밖에 완전히 배치하는 것을 허용하지 않습니다. 화면의 오른쪽 또는 왼쪽 하단 모서리에 1픽셀 세로줄의 "숨겨진" 창이 계속 표시됩니다. 즉, AeroSpace가 심하게 충돌하는 경우에도 이 몇 픽셀을 화면 중앙으로 드래그하여 창을 수동으로 "숨기기 해제"할 수 있습니다.

숨겨진 창의 표시를 최소화하려면 Dock을 하단에 배치하고 자동 숨김 기능을 켜는 것이 좋습니다.

4.1. 올바른 모니터 배치

AeroSpace는 창을 숨길 수 있는 여유 공간이 필요하므로 모든 모니터의 오른쪽 또는 왼쪽 하단에 여유 공간이 있도록 모니터를 배치하시기 바랍니다. (System Settings → Displays → Arrange…​)

모니터를 올바르게 배치하지 않으면 숨겨진 창의 일부가 다른 모니터에 표시됩니다.

그림 5. 잘못된 모니터 배열. 모니터 2의 하단 모서리에 여유 공간이 없습니다.

그림 6. 올바른 모니터 배치. 모든 모니터의 하단 모서리에 여유 공간이 있습니다.

그림 7. 잘못된 모니터 배치. 모니터 1의 하단 모서리에 여유 공간이 없습니다.

그림 8. 양호한 모니터 배치. 모든 모니터의 하단 모서리에 여유 공간이 있습니다.

4.2. 미션 컨트롤에 대한 참고 사항

어떤 이유에서인지 미션 컨트롤에서 화면 오른쪽 하단에 많은 창을 배치하는 것을 좋아하지 않습니다. 창을 더 크게 표시할 공간이 충분한데도 미션 컨트롤에서 창을 너무 작게 표시합니다.

해결 방법이 있습니다. Group windows by application 설정을 활성화할 수 있습니다:

defaults write com.apple.dock expose-group-apps -bool true && killall Dock

(또는 시스템 설정: System Settings → Desktop & Dock → Group windows by application)에서 설정할 수 있습니다. 어떤 이상한 이유에서든 도움이 됩니다.

4.3. '디스플레이에 별도의 스페이스가 있음'에 대한 참고 사항

이 설정을 비활성화하면 macOS가 더 안정적으로 작동한다는 관찰 결과가 있습니다 Displays have separate Spaces. (기본적으로 활성화되어 있습니다.) 이 설정을 활성화하면 포커스 및 성능과 관련된 모든 종류의 이상한 문제를 보고하는 사람들이 있습니다:

  • 멀티 모니터 설정에서 잘못된 창이 포커스를 받을 수 있습니다: #101 (Apple API의 버그)

  • 단일 모니터 설정에서 테두리가 없는 잘못된 Alacritty 창이 포커스를 받을 수 있습니다: #247 (Apple API의 버그)

  • 성능 문제: #333

  • macOS가 무작위로 초점을 다시 전환합니다: #289

Displays have separate Spaces 을 활성화하면 모니터 간에 창을 이동하면 창이 다른 스페이스 간에 이동하게 되는데, 이는 AeroSpace가 사용하는 공용 API가 스페이스의 존재를 인식하지 못하기 때문입니다. 스페이스는 macOS에서 저주받은 존재일 뿐입니다. 스페이스가 적을수록 macOS가 더 잘 작동합니다.

'디스플레이에 별도의 스페이스가 있음'이 활성화된 경우 '디스플레이에 별도의 스페이스가 있음'이 비활성화됨

창을 여러 모니터에 걸쳐 사용할 수 있나요?

❌ 아니요. macOS 제한 사항

👍 예

❌ 이상한 초점 및 성능 문제가 발생할 수 있습니다(위 목록 참조).

👍 공용 Apple API가 더 안정적입니다(이는 AeroSpace 안정성에 영향을 미침).

첫 번째 모니터가 전체 화면일 때

두 번째 모니터가 독립적으로 작동

❌ 두 번째 모니터가 사용할 수 없는 검은색 화면인 경우

macOS 상태 표시줄 ...

...이 두 모니터에 모두 표시됨

...이 메인 모니터에만 표시됨

멀티 모니터 설정에서 macOS 기본 전체 화면에 신경 쓰지 않는 경우 (별도의 Space 인스턴스를 생성하기 때문에 어쨌든 그 자체가 투박함) Displays have separate Spaces 을 비활성화하는 것이 좋습니다.

이 설정은 실행하여 비활성화할 수 있습니다:

defaults write com.apple.spaces spans-displays -bool true && killall SystemUIServer

(또는 시스템 설정: System Settings → Desktop & Dock → Displays have separate Spaces)에서 비활성화할 수 있습니다. 설정을 적용하려면 로그아웃해야 합니다.

5. 콜백

5.1. '창 감지 시' 콜백

on-window-detected 콜백을 사용하여 새 창이 감지될 때마다 명령을 실행할 수 있습니다.

다음은 가능한 모든 구성을 사용하는 쇼케이스 예시입니다:

[[on-window-detected]]
    if.app-id = 'com.apple.systempreferences'
    if.app-name-regex-substring = 'settings'
    if.window-title-regex-substring = 'substring'
    if.workspace = 'workspace-name'
    if.during-aerospace-startup = true
    check-further-callbacks = true
    run = ['layout floating', 'move-node-to-workspace S']  # The callback itself

run 감지된 창이 지정된 모든 조건과 일치하는 경우에만 명령이 실행됩니다. 조건을 지정하지 않으면 새 창이 감지될 때마다 run 이 실행됩니다.

구성에서 여러 콜백을 선언할 수 있습니다. 콜백은 선언된 순서대로 처리됩니다. 기본적으로 조건과 일치하는 첫 번째 콜백이 실행되고 추가 콜백은 고려되지 않습니다. ( check-further-callbacks 옵션으로 동작을 재정의할 수 있습니다.)

현재로서는 run 콜백에서 move-node-to-workspace 및 레이아웃 명령만 지원됩니다. 다른 명령이 지원되도록 하려면 사용 사례를 이슈에 게시해 주세요.

사용 가능한 창 조건은 다음과 같습니다:

조건 TOML 키 조건 설명

if.app-id

애플리케이션 ID가 감지된 창과 정확히 일치

if.app-name-regex-substring

애플리케이션 이름 감지된 창의 대소문자를 구분하지 않는 정규식 하위 문자열

if.window-title-regex-substring

감지된 창의 창 제목 대소문자를 구분하지 않는 정규식 부분 문자열

if.during-aerospace-startup

  • true 이면 AeroSpace 시작 중에만 콜백을 실행합니다.

  • false 이면 AeroSpace 시작 중이 아닌 경우에만 콜백을 실행합니다.

  • 지정하지 않으면 조건이 확인되지 않습니다.

if.workspace

창의 작업 공간 이름 정확히 일치

  • if.during-aerospace-startup = true 은 시작 시에만 초기 앱 배열을 수행하려는 경우에 유용합니다.

  • if.during-aerospace-startup = false 는 AeroSpace를 다시 시작하고 싶지만 콜백이 다시 시작할 때마다 실행하고 싶지 않은 부작용이 있는 경우 유용합니다(예: 콜백이 새 창에서 열림).

app-id 을 확인하는 방법에는 여러 가지가 있습니다:

  • 인기 있는 애플리케이션 ID의 미리 구성된 목록을 살펴보세요.

  • 다음을 사용할 수 있습니다. aerospace list-apps CLI 명령을 사용하여 실행 중인 애플리케이션의 ID 가져오기

  • mdls -name kMDItemCFBundleIdentifier -r /Applications/App.app

중요

 일부 창은 창이 나타난 후 제목을 초기화합니다.window-title-regex-substring 은 이러한 창에서 예상대로 작동하지 않을 수 있습니다.

자동화 예시:

  • 특정 작업 공간에 앱 할당

    [[on-window-detected]]
    if.app-id = 'org.alacritty'
    run = 'move-node-to-workspace T' # mnemonics T - Terminal

[[on-window-detected]]
    if.app-id = 'com.google.Chrome'
    run = 'move-node-to-workspace W' # mnemonics W - Web browser

[[on-window-detected]]
    if.app-id = 'com.jetbrains.intellij'
    run = 'move-node-to-workspace I' # mnemonics I - IDE

  • 모든 창을 기본적으로 부동 상태로 만들기

    [[on-window-detected]]
    check-further-callbacks = true
    run = 'layout floating'

5.2. '포커스 변경 시' 콜백

다음 콜백을 사용하여 포커스 변경을 추적할 수 있습니다: on-focus-changed 및 on-focused-monitor-changed.

  • on-focus-changed 는 초점이 맞춰진 창이나 작업 영역이 변경될 때마다 호출됩니다.

  • on-focused-monitor-changed 는 초점이 맞춰진 모니터가 변경될 때마다 호출됩니다.

콜백의 일반적인 사용 사례는 "마우스가 포커스를 따르는" 동작을 구현하는 것입니다. 원하는 콜백을 마우스 이동 명령과 결합하기만 하면 됩니다:

on-focused-monitor-changed = ['move-mouse monitor-lazy-center'] # Mouse lazily follows focused monitor (default in i3)
# or
on-focus-changed = ['move-mouse window-lazy-center'] # Mouse lazily follows any focus (window or workspace)

콜백이 호출되는 순서는 구현 세부 사항이며 버전마다 변경될 수 있으므로 콜백이 호출되는 순서에 의존해서는 안 됩니다.

콜백은 "재귀 저항성"이 있으므로 콜백 내에서 포커스를 변경해도 콜백이 다시 트리거되지 않습니다. 이러한 콜백 내에서 포커스를 변경하는 것은 좋지 않은 생각이며 향후 버전에서는 처리 방식이 변경될 가능성이 높습니다.

5.3. 'exec-on-workspace-change' 콜백

exec-on-workspace-change 콜백을 사용하면 포커스된 작업 영역이 변경될 때 임의의 프로세스를 실행할 수 있습니다. 바를 통합하는 데 유용할 수 있습니다.

# Notify Sketchybar about workspace change
exec-on-workspace-change = ['/bin/bash', '-c',
    'sketchybar --trigger aerospace_workspace_change FOCUSED=$AEROSPACE_FOCUSED_WORKSPACE'
]

프로세스는 exec.env-vars프로세스에는 다음 환경 변수에 대한 액세스 권한이 있습니다:

  • AEROSPACE_FOCUSED_WORKSPACE - 사용자가 전환한 워크스페이스

  • AEROSPACE_PREV_WORKSPACE - 에서 전환한 작업 공간 사용자

Sketchybar와 통합하는 방법에 대한 자세한 예는./goodies를 참조하세요.

6. 여러 모니터

  • 작업 공간 풀은 모니터 간에 공유됩니다.

  • 각 모니터는 자체 작업 공간을 표시합니다. 표시된 작업 공간을 "보이는" 작업 공간이라고 합니다.

  • 서로 다른 모니터는 같은 작업 공간을 동시에 표시할 수 없습니다.

  • 각 워크스페이스(보이지 않거나 비어 있더라도)에는 모니터가 할당되어 있습니다.

  • 기본적으로 모든 워크스페이스는 '메인' 모니터(' System → Displays → Use as'의 '메인')에 할당됩니다.

작업 공간으로 전환하면

  1. AeroSpace가 작업 공간의 할당된 모니터를 가져와 모니터에 작업 공간을 표시합니다.

  2. AeroSpace가 작업 공간에 초점을 맞춥니다.

작업 공간 풀을 공유하는 아이디어는 대부분의 사용자가 보조 모니터에 제한된 작업 공간 세트를 가지고 있다는 관찰에 기반합니다. 보조 모니터는 특정 작업(브라우저, 셸) 전용이거나 로그 및 대시보드와 같은 다양한 활동을 모니터링하는 데 자주 사용됩니다. 따라서 각 보조 모니터당 하나의 작업 공간을 사용하고 "나머지"는 메인 모니터에서 사용하는 것이 합리적일 때가 많습니다.

참고

빈 작업 공간으로 전환하는 것이 AeroSpace와 i3의 유일한 차이점입니다. 빈 작업 공간으로 전환하면 AeroSpace는 할당된 모니터에 작업 공간을 배치하고, i3는 현재 활성 상태인 모니터에 작업 공간을 배치합니다.

  • 위에 나열된 관찰을 통해 AeroSpace 모델이 더 잘 작동한다는 것을 알았습니다.

  • AeroSpace 모델이 더 일관성이 있습니다(빈 작업 공간과 비어 있지 않은 작업 공간에 대해 동일하게 작동).

6.1. 모니터에 작업 공간 할당하기

workspace-to-monitor-force-assignment 구문을 사용하여 특정 모니터에 항상 표시되도록 작업 공간을 할당할 수 있습니다.

[workspace-to-monitor-force-assignment]
    1 = 1                            # Monitor sequence number from left to right. 1-based indexing
    2 = 'main'                       # Main monitor
    3 = 'secondary'                  # Non-main monitor in case when there are only two monitors
    4 = 'built-in'                   # Case insensitive regex substring
    5 = '^built-in retina display$'  # Case insensitive regex match
    6 = ['secondary', 'dell']        # You can specify multiple patterns.
                                     #   The first matching pattern will be used

  • 할당의 왼쪽은 작업 공간 이름입니다.

  • 할당의 오른쪽은 모니터 패턴입니다.

지원되는 모니터 패턴

  • main - "메인" 모니터( System Settings → Displays → Use as 에서 "메인")

  • secondary - 모니터가 두 개만 있는 경우 주 모니터가 아닌 모니터

  • <number> (예: 1, 2) - 왼쪽에서 오른쪽으로 모니터의 시퀀스 번호입니다. 1 기반 인덱싱

  • <regex-pattern> (예: dell.*, built-in.*) - 대소문자를 구분하지 않는 정규식 하위 문자열 패턴

여러 패턴을 배열로 지정할 수 있습니다. 첫 번째 일치하는 패턴이 사용됩니다.

7. 대화 상자 휴리스틱

  • Apple은 앱이 어떤 창이 대화상자인지 다른 사용자에게 알릴 수 있도록 접근성 API를 제공합니다.

  • 많은 앱이 이 API를 구현하지 않거나 부적절하게 구현합니다.

    일부 Apple 대화 상자조차도 API를 제대로 구현하지 않습니다. (예: Finder "복사" 진행률 창은 다른 사람에게 대화 상자라는 것을 알리지 않습니다.)

AeroSpace는 API를 사용하여 창이 대화상자인지 여부를 부드럽게 묻지만, 몇 가지 휴리스틱을 적용하기도 합니다.

예를 들어 전체 화면 버튼이 없는 창(NB! 전체 화면 버튼과 최대화 버튼은 다른 버튼입니다)은 일반적으로 대화 상자로 간주되며, 터미널 앱(WezTerm, Alacritty, iTerm2 등)은 제외됩니다.

대화 상자로 인식되는 창은 기본적으로 떠 있습니다.

일부 창이 제대로 처리되지 않는 경우 휴리스틱을 개선하는 PR을 만들어도 좋습니다. 자주 사용하는 애플리케이션에 대한 특수 처리를 하드코딩하는 것은 괜찮습니다. AeroSpace 소스에서 isDialogHeuristic 함수를 참조하세요.

on-window-detected 을 사용하여 특정 애플리케이션의 모든 창을 강제 타일링하거나 강제 플로팅할 수도 있습니다:

  1. 모든 창(또는 특정 앱의 창) 강제 타일링하기

    [[on-window-detected]]
    if.app-id = '...'
    run = 'layout tiling'

  2. 모든 창(또는 특정 앱의 창) 강제 플로팅

    [[on-window-detected]]
    if.app-id = '...'
    run = 'layout floating'

8. 일반적인 함정: 키보드 키 처리

구성에서 AeroSpace가 일부 키를 처리할 수 없는 경우, 전역 키를 수신할 수 있는 다른 소프트웨어(예: skhd, Karabiner-Elements, Raycast)와 키가 충돌하지 않는지 확인하세요.
