앤트로픽의 엔지니어링

최근 에이전트 코딩을 위한 명령줄 도구인 Claude Code를 출시했습니다. 연구 프로젝트로 개발된 Claude Code는 Anthropic 엔지니어와 연구원이 Claude를 코딩 워크플로우에 통합할 수 있는 보다 네이티브한 방법을 제공합니다.

Claude Code는 의도적으로 로우레벨이며 의견이 개입되지 않아 특정 워크플로를 강요하지 않고 원시 모델에 가까운 액세스를 제공합니다. 이러한 설계 철학은 유연하고 커스터마이징이 가능하며 스크립팅이 가능하고 안전한 파워 툴을 만들어냅니다. 이러한 유연성은 강력하지만 에이전트 코딩 툴을 처음 접하는 엔지니어에게는 적어도 자신만의 모범 사례를 개발할 때까지는 학습 곡선이 존재합니다.

이 게시물에서는 다양한 코드베이스, 언어 및 환경에서 Claude Code를 사용하는 Anthropic 내부 팀과 외부 엔지니어 모두에게 효과적인 것으로 입증된 일반적인 패턴을 간략하게 설명합니다. 이 목록의 어떤 것도 정해진 것이 아니며 보편적으로 적용 가능한 것도 아니므로 이 제안을 시작점으로 삼으세요. 실험을 통해 자신에게 가장 적합한 방법을 찾아보시기 바랍니다!

더 자세한 정보를 찾고 계신가요? claude.ai/code의 포괄적인 문서에서 이 게시물에 언급된 모든 기능을 다루고 있으며 추가 예제, 구현 세부 사항 및 고급 기술을 제공합니다.

1. 설정 사용자 지정

Claude Code는 프롬프트에 컨텍스트를 자동으로 가져오는 에이전트 코딩 도우미입니다. 이 컨텍스트 수집에는 시간과 토큰이 소모되지만 환경 조정을 통해 이를 최적화할 수 있습니다.

a. CLAUDE.md 파일 생성

CLAUDE.md 은 대화를 시작할 때 Claude가 자동으로 컨텍스트를 가져오는 특수 파일입니다. 따라서 문서화를 위한 이상적인 장소입니다:

  • 일반적인 bash 명령
  • 핵심 파일 및 유틸리티 함수
  • 코드 스타일 가이드라인
  • 테스트 지침
  • 리포지토리 에티켓(예: 브랜치 이름 지정, 병합 대 리베이스 등)
  • 개발자 환경 설정(예: pyenv 사용, 어떤 컴파일러가 작동하는지 등)
  • 프로젝트와 관련된 예기치 않은 동작 또는 경고
  • Claude가 기억했으면 하는 기타 정보

CLAUDE.md 파일에는 필수 형식이 없습니다. 간결하고 사람이 읽기 쉽게 작성하는 것이 좋습니다. 예를 들어

# Bash commands
- npm run build: Build the project
- npm run typecheck: Run the typechecker

# Code style
- Use ES modules (import/export) syntax, not CommonJS (require)
- Destructure imports when possible (eg. import { foo } from 'bar')

# Workflow
- Be sure to typecheck when you’re done making a series of code changes
- Prefer running single tests, and not the whole test suite, for performance

CLAUDE.md 파일을 여러 위치에 배치할 수 있습니다:

  • 리포지토리의루트, 또는 claude 을 실행하는 위치(가장 일반적인 용도). 이름을 CLAUDE.md 으로 지정하고 git에 체크인하여 여러 세션 및 팀과 공유할 수 있도록 하거나(권장), CLAUDE.local.md 및 .gitignore 으로 지정할 수 있습니다.
  • 을 실행하는디렉터리의 모든 상위 claude. 이는 root/foo 에서 claude 을 실행하고 root/CLAUDE.md 과 root/foo/CLAUDE.md 모두에 CLAUDE.md 파일이 있는 모노리포지션에 가장 유용하다. 이 두 파일은 자동으로 컨텍스트로 가져옵니다.
  • claude 을 실행하는디렉터리의 모든 하위 디렉터리. 이것은 위와 반대이며, 이 경우 Claude는 하위 디렉터리의 파일로 작업할 때 필요에 따라 CLAUDE.md 파일을 가져옵니다.
  • 홈 폴더 (~/.claude/CLAUDE.md)는 모든 클라우데 세션에 적용됩니다.

/init 명령을 실행하면 Claude가 자동으로 CLAUDE.md 을 생성합니다.

b. CLAUDE.md 파일 조정하기

CLAUDE.md 파일은 Claude의 프롬프트의 일부가 되므로 자주 사용하는 프롬프트처럼 다듬어야 합니다. 흔히 저지르는 실수는 효과에 대한 반복적인 검토 없이 광범위한 콘텐츠를 추가하는 것입니다. 시간을 들여 실험해 보고 모델에서 어떤 것이 가장 좋은 안내를 생성하는지 결정하세요.

CLAUDE.md 에 콘텐츠를 수동으로 추가하거나 # 키를 눌러 관련 CLAUDE.md 에 자동으로 통합되도록 클로드에게 지시를 내릴 수 있습니다. 많은 엔지니어가 # 를 자주 사용하여 코딩하는 동안 명령어, 파일 및 스타일 가이드라인을 문서화한 다음 CLAUDE.md 변경 사항을 커밋에 포함시켜 팀원들도 혜택을 볼 수 있도록 합니다.

Anthropic에서는 때때로 CLAUDE.md 파일을 프롬프트 개선기를 통해 실행하고 지침을 조정하여(예: "중요" 또는 "반드시"로 강조 표시) 준수도를 높입니다.

c. Claude의 허용 도구 목록 큐레이팅하기

기본적으로 Claude Code는 파일 쓰기, 여러 bash 명령, MCP 도구 등 시스템을 수정할 수 있는 모든 작업에 대한 권한을 요청합니다. 저희는 안전을 우선시하기 위해 의도적으로 보수적인 접근 방식으로 Claude Code를 설계했습니다. 허용 목록을 사용자 지정하여 안전하다고 판단되는 추가 도구를 허용하거나 잠재적으로 안전하지 않은 도구(예: 파일 편집, git commit)는 쉽게 실행 취소할 수 있도록 허용 목록을 사용자 지정할 수 있습니다.

허용된 도구를 관리하는 방법에는 네 가지가 있습니다:

  • 세션 중 메시지가 표시되면'항상 허용'을 선택합니다.
  • 클로드 코드를 시작한 후 /permissions 명령을 사용하여 허용 목록에서 도구를 추가하거나 제거합니다. 예를 들어 Edit 을 추가하여 파일 편집을 항상 허용하거나 Bash(git commit:*) 을 추가하여 git 커밋을 허용하거나 mcp__puppeteer__puppeteer_navigate 을 추가하여 Puppeteer MCP 서버로 탐색을 허용할 수 있습니다.
  • .claude/settings.json 또는 ~/.claude.json 을수동으로 편집하세요(팀과 공유하려면 전자를 소스 제어에 체크하는 것이 좋습니다).
  • 세션별 권한은 --allowedTools CLI 플래그를 사용하세요.

d. GitHub를 사용하는 경우 gh CLI를 설치합니다.

Claude는 gh CLI를 사용하여 이슈 생성, 풀 리퀘스트 열기, 댓글 읽기 등을 위해 GitHub와 상호 작용하는 방법을 알고 있습니다. gh 을 설치하지 않아도 Claude는 GitHub API 또는 MCP 서버(설치되어 있는 경우)를 계속 사용할 수 있습니다.

2. Claude에게 더 많은 도구 제공

Claude는 셸 환경에 액세스할 수 있으며, 여기서 사용자가 직접 사용하는 것처럼 편리한 스크립트 및 함수 세트를 만들 수 있습니다. 또한 MCP 및 REST API를 통해 더 복잡한 도구를 활용할 수도 있습니다.

a. bash 도구와 함께 Claude 사용

Claude 코드는 사용자의 bash 환경을 상속하여 모든 도구에 액세스할 수 있습니다. Claude는 유닉스 도구 및 gh 와 같은 일반적인 유틸리티는 알고 있지만, 사용자 지정 bash 도구는 지시 없이는 알지 못합니다:

  1. Claude에게 사용 예제와 함께 도구 이름을 알려주세요.
  2. Claude에게 --help 을 실행하여 도구 설명서를 보라고 합니다.
  3. 자주 사용하는 도구를 CLAUDE.md

b. MCP와 함께 Claude 사용

클로드 코드는 MCP 서버와 클라이언트로 작동합니다. 클라이언트로서 세 가지 방법으로 MCP 서버에 연결하여 해당 도구에 액세스할 수 있습니다:

  • 프로젝트 구성에서 (해당 디렉토리에서 클로드 코드를 실행할 때 사용 가능)
  • 글로벌 설정에서 (모든 프로젝트에서 사용 가능)
  • 체크인된 .mcp.json 파일 (코드베이스에서 작업하는 모든 사람이 사용 가능). 예를 들어 .mcp.json 에 Puppeteer 및 Sentry 서버를 추가하여 리포지토리에서 작업하는 모든 엔지니어가 즉시 사용할 수 있도록 할 수 있습니다.

MCP로 작업할 때 구성 문제를 식별하는 데 도움이 되도록 --mcp-debug 플래그를 사용하여 Claude를 실행하는 것도 도움이 될 수 있습니다.

c. 사용자 지정 슬래시 명령 사용

반복되는 워크플로우(디버깅 루프, 로그 분석 등)의 경우 .claude/commands 폴더에 있는 마크다운 파일에 프롬프트 템플릿을 저장하세요. 이러한 템플릿은 / 을 입력하면 슬래시 명령 메뉴를 통해 사용할 수 있습니다. 이러한 명령을 git에 체크인하여 다른 팀원들이 사용할 수 있도록 할 수 있습니다.

사용자 지정 슬래시 명령에는 특수 키워드 $ARGUMENTS 를 포함시켜 명령 호출에서 매개변수를 전달할 수 있습니다.

예를 들어 다음은 Github 이슈를 자동으로 가져와서 수정하는 데 사용할 수 있는 슬래시 명령어입니다:

Please analyze and fix the GitHub issue: $ARGUMENTS.

Follow these steps:

1. Use `gh issue view` to get the issue details
2. Understand the problem described in the issue
3. Search the codebase for relevant files
4. Implement the necessary changes to fix the issue
5. Write and run tests to verify the fix
6. Ensure code passes linting and type checking
7. Create a descriptive commit message
8. Push and create a PR

Remember to use the GitHub CLI (`gh`) for all GitHub-related tasks.

위의 내용을 .claude/commands/fix-github-issue.md 에 넣으면 Claude Code에서 /project:fix-github-issue 명령으로 사용할 수 있습니다. 그런 다음 예를 들어 /project:fix-github-issue 1234 을 사용하여 Claude가 이슈 #1234를 수정하도록 할 수 있습니다. 마찬가지로 모든 세션에서 사용하려는 명령에 대해 ~/.claude/commands 폴더에 자신만의 개인 명령을 추가할 수 있습니다.

3. 일반적인 워크플로 시도하기

클로드 코드는 특정 워크플로우를 강요하지 않으므로 원하는 방식으로 유연하게 사용할 수 있습니다. 이러한 유연성 덕분에 사용자 커뮤니티에서 Claude Code를 효과적으로 사용하는 몇 가지 성공적인 패턴이 등장했습니다:

a. 탐색, 계획, 코딩, 커밋

이 다용도 워크플로는 많은 문제에 적합합니다:

  1. 일반적인 포인터("로깅을 처리하는 파일 읽기") 또는 특정 파일 이름("logging.py 읽기")을 제공하여 관련 파일, 이미지 또는 URL을 읽도록 Claude에게 요청하되 아직 코드를 작성하지 않도록 명시적으로 지시합니다.
    1. 특히 복잡한 문제에 대해 서브에이전트의 강력한 사용을 고려해야 하는 워크플로우의 일부입니다. 특히 대화나 작업 초기에 세부 사항을 확인하거나 특정 질문을 조사할 때 하위 에이전트를 사용하도록 Claude에게 지시하면 효율성 측면에서 큰 단점 없이 컨텍스트 가용성을 유지하는 경향이 있습니다.
  2. Claude에게 특정 문제에 접근하는 방법에 대한 계획을 세우도록 요청하세요. "생각하다"라는 단어를 사용하여 확장 사고 모드를 트리거하면 Claude가 대안을 더 철저하게 평가할 수 있는 추가 계산 시간을 확보할 수 있습니다. 이러한 특정 문구는 시스템에서 사고 예산의 증가 수준에 직접 매핑됩니다: "생각하다" < "열심히 생각하다" < "더 열심히 생각하다" < "울트라씽크." 각 레벨은 클로드가 사용할 수 있는 사고 예산을 점진적으로 더 많이 할당합니다.
    1. 이 단계의 결과가 합리적이라고 생각되면 Claude에게 그 계획이 담긴 문서나 GitHub 이슈를 만들게 하여 구현(3단계)이 원하는 대로 되지 않을 경우 이 지점에서 다시 시작할 수 있도록 할 수 있습니다.
  3. Claude에게 솔루션을 코드로 구현해 달라고 요청하세요. 이 단계는 솔루션의 일부를 구현하면서 솔루션의 합리성을 명시적으로 검증하도록 요청하기에 좋은 곳이기도 합니다.
  4. Claude에게 결과를 커밋하고 풀 리퀘스트를 만들어 달라고 요청하세요. 관련성이 있는 경우, 이 단계는 클로드가 방금 수행한 작업에 대한 설명과 함께 README 또는 변경 로그를 업데이트하도록 하는 좋은 시기이기도 합니다.

1번~2번 단계는 매우 중요하며, 이 단계가 없으면 Claude는 곧바로 솔루션 코딩으로 넘어가는 경향이 있습니다. 때로는 그렇게 하는 것이 좋지만, Claude에게 먼저 조사하고 계획하도록 요청하면 사전에 더 깊이 생각해야 하는 문제의 경우 성능이 크게 향상됩니다.

b. 테스트 작성, 커밋; 코딩, 반복, 커밋

단위, 통합 또는 엔드투엔드 테스트를 통해 쉽게 검증할 수 있는 변경 사항에 대해 Anthropic이 선호하는 워크플로입니다. 에이전트 코딩을 사용하면 테스트 중심 개발(TDD)이 더욱 강력해집니다:

  1. Claude에게 예상되는 입력/출력 쌍을 기반으로 테스트를 작성해 달라고 요청하세요. 코드베이스에 아직 존재하지 않는 기능에 대해서도 모의 구현을 만들지 않도록 테스트 중심 개발을 하고 있다는 사실을 명시하세요.
  2. Claude에게 테스트를 실행하고 실패를 확인하도록 하세요. 이 단계에서는 구현 코드를 작성하지 말라고 명시적으로 지시하는 것이 도움이 되는 경우가 많습니다.
  3. 테스트가 만족스러우면클로드에게 커밋해달라고 요청하세요 .
  4. 클로드에게 테스트를 통과하는 코드를 작성하도록 요청하면서 테스트를 수정하지 않도록 지시합니다. 클로드에게 모든 테스트가 통과될 때까지 계속 진행하라고 말합니다. 일반적으로 Claude가 코드를 작성하고, 테스트를 실행하고, 코드를 조정하고, 테스트를 다시 실행하는 데 몇 번의 반복이 필요합니다.
    1. 이 단계에서는 구현이 테스트에 과도하게 적합하지 않은지 독립적인 하위 에이전트로 확인하도록 요청하는 것이 도움이 될 수 있습니다.
  5. 변경 사항이 만족스러우면Claude에게 코드를 커밋하도록 요청하세요.

Claude는 시각적 모의, 테스트 케이스 또는 다른 종류의 결과물 등 반복할 명확한 목표가 있을 때 가장 잘 작동합니다. 테스트와 같은 예상 출력을 제공함으로써 Claude는 변경을 수행하고 결과를 평가하며 성공할 때까지 점진적으로 개선할 수 있습니다.

c. 코드 작성, 결과 스크린샷, 반복

테스트 워크플로와 마찬가지로 Claude에게 시각적 목표를 제공할 수 있습니다:

  1. Claude에게 브라우저 스크린샷을 찍을 수 있는 방법을 제공합니다 (예: Puppeteer MCP 서버, iOS 시뮬레이터 MCP 서버 또는 수동으로 스크린샷을 복사하여 Claude에 붙여넣기).
  2. 이미지를 복사/붙여넣기 또는 드래그 앤 드롭하거나 이미지 파일 경로를 알려주어 Claude에게시각적 모형을 제공합니다.
  3. Claude에게 디자인을 코드로구현하고 결과의 스크린샷을 찍어 모의 결과와 일치할 때까지 반복해 달라고 요청하세요.
  4. 만족스러우면 클로드에게커밋을 요청 하세요.

인간과 마찬가지로 클로드의 결과물은 반복을 통해 크게 개선되는 경향이 있습니다. 첫 번째 버전도 괜찮을 수 있지만 일반적으로 2~3번 반복하면 훨씬 더 좋아집니다. 최상의 결과를 위해 Claude에게 결과물을 확인할 수 있는 도구를 제공하세요.

d. 안전한 YOLO 모드

Claude를 감독하는 대신 claude --dangerously-skip-permissions 을 사용하여 모든 권한 검사를 우회하고 완료될 때까지 Claude가 중단 없이 작업하도록 할 수 있습니다. 이 모드는 린트 오류 수정이나 상용구 코드 생성과 같은 워크플로우에 적합합니다.

Claude가 임의의 명령을 실행하도록 하는 것은 위험하며 데이터 손실, 시스템 손상 또는 데이터 유출(예: 프롬프트 인젝션 공격)을 초래할 수 있습니다. 이러한 위험을 최소화하려면 인터넷에 액세스할 수 없는 컨테이너에서 --dangerously-skip-permissions 을 사용하세요. Docker 개발 컨테이너를 사용하여 이 참조 구현을 따를 수 있습니다.

e. 코드베이스 Q&A

새 코드베이스로 온보딩할 때는 학습 및 탐색을 위해 Claude Code를 사용하세요. 쌍 프로그래밍을 할 때 프로젝트의 다른 엔지니어에게 질문하는 것과 동일한 종류의 질문을 Claude에게 할 수 있습니다. Claude는 코드베이스를 에이전트 방식으로 검색하여 다음과 같은 일반적인 질문에 답할 수 있습니다:

  • 로깅은 어떻게 작동하나요?
  • 새 API 엔드포인트를 만들려면 어떻게 해야 하나요?
  • foo.rs 의 134번 라인에서 async move { ... } 은 무엇을 하나요?
  • CustomerOnboardingFlowImpl 은 어떤 엣지 케이스를 처리하나요?
  • 333줄에서 bar() 대신 foo() 을 호출하는 이유는 무엇인가요?
  • Java에서 baz.py 의 334 줄에 해당하는 것은 무엇인가요?

앤트로픽에서는 이러한 방식으로 Claude Code를 사용하는 것이 핵심 온보딩 워크플로우가 되어 램프업 시간을 크게 개선하고 다른 엔지니어의 부하를 줄였습니다. 특별한 프롬프트가 필요하지 않습니다! 질문만 하면 Claude가 코드를 탐색하여 답을 찾아줍니다.

f. Claude를 사용하여 git과 상호 작용

Claude는 많은 git 작업을 효과적으로 처리할 수 있습니다. 많은 Anthropic 엔지니어가 90% 이상의 git 상호 작용에 Claude를 사용합니다:

  • "어떤 변경 사항이 v1.2.3으로 변경되었나?", "이 특정 기능의 소유자는 누구인가?", "이 API가 왜 이런 식으로 설계되었나?" 등의 질문에 답하기 위해 깃 기록을 검색합니다. 이와 같은 쿼리에 대한 답을 찾기 위해 Claude에게 명시적으로 커밋 히스토리를 살펴보도록 유도하는 것이 도움이 됩니다.
  • 커밋 메시지 작성: Claude는 변경 사항과 최근 기록을 자동으로 살펴보고 모든 관련 컨텍스트를 고려하여 메시지를 작성합니다.
  • 파일 되돌리기, 리베이스 충돌 해결, 패치 비교 및 접목과 같은복잡한 Git 작업 처리하기

g. Claude를 사용하여 GitHub와 상호 작용하기

Claude Code는 다양한 GitHub 상호작용을 관리할 수 있습니다:

  • 풀 리퀘스트 생성: Claude는 약어 "pr"을 이해하고 차이점 및 주변 컨텍스트에 따라 적절한 커밋 메시지를 생성합니다.
  • 간단한 코드 리뷰 댓글에 대한원샷 해결 구현: PR에 대한 댓글을 수정하라고 지시하고(선택 사항으로 더 구체적인 지침을 제공) 완료되면 PR 브랜치로 푸시백하세요.
  • 실패한 빌드 또는 린터 경고수정하기
  • Claude에게 열려 있는 GitHub 이슈를 반복하도록 요청하여 열려 있는 이슈를분류하고 분류하기

이렇게 하면 일상적인 작업을 자동화하는 동안 gh 명령줄 구문을 기억할 필요가 없습니다.

h. Claude를 사용해 Jupyter 노트북으로 작업하기

Anthropic의 연구원과 데이터 과학자들은 Claude 코드를 사용해 Jupyter 노트북을 읽고 씁니다. Claude는 이미지를 포함한 결과물을 해석할 수 있어 데이터를 빠르게 탐색하고 상호 작용할 수 있는 방법을 제공합니다. 필수 프롬프트나 워크플로는 없지만, VS Code에서 Claude Code와 .ipynb 파일을 나란히 열어 두는 워크플로를 권장합니다.

또한 동료들에게 보여주기 전에 Claude에게 Jupyter 노트북을 정리하거나 미적으로 개선해 달라고 요청할 수도 있습니다. 노트북이나 데이터 시각화를 "미적으로 보기 좋게" 만들라고 구체적으로 지시하면 사람이 보는 환경에 최적화되어 있다는 사실을 상기시키는 데 도움이 됩니다.

4. 워크플로우 최적화

아래 제안 사항은 모든 워크플로우에 적용됩니다:

a. 지침을 구체적으로 작성하세요

특히 첫 번째 시도에서 보다 구체적인 지침을 제공하면 클로드 코드의 성공률이 크게 향상됩니다. 미리 명확한 지침을 제공하면 나중에 코스를 수정할 필요성이 줄어듭니다.

예를 들어

PoorGood
foo.py에 대한 테스트 추가에 대한 새 테스트 케이스를 작성하여 사용자가 로그아웃한 에지 케이스를 다루고 모의 테스트를 피하세요.
ExecutionFactory에는 왜 이렇게 이상한 API가 있을까요?ExecutionFactory의 git 히스토리를 살펴보고 그 API가 어떻게 만들어졌는지 요약해 보세요.
캘린더 위젯 추가하기기존 위젯이 홈페이지에서 어떻게 구현되었는지 살펴보고 패턴과 특히 코드와 인터페이스가 어떻게 분리되어 있는지 이해합니다. 핫도그위젯.php는 시작하기에 좋은 예시입니다. 그런 다음 패턴을 따라 사용자가 월을 선택하고 앞뒤로 페이지 넘기기를 통해 연도를 선택할 수 있는 새로운 캘린더 위젯을 구현하세요. 나머지 코드베이스에서 이미 사용된 라이브러리 이외의 라이브러리 없이 처음부터 빌드하세요.

Claude는 의도를 유추할 수는 있지만 마음을 읽을 수는 없습니다. 구체적일수록 기대치에 더 잘 부합합니다.

b. Claude에게 이미지 제공

Claude는 여러 가지 방법을 통해 이미지와 다이어그램에 탁월한 능력을 발휘합니다:

  • 스크린샷 붙여넣기 (전문가 팁: macOS에서는 cmd+ctrl+shift+4를 눌러 스크린샷을 클립보드에 복사하고 ctrl+v를 눌러 붙여넣으세요. 이 방법은 Mac에서 일반적으로 붙여넣을 때 사용하는 cmd+v가 아니며 원격으로 작동하지 않는다는 점에 유의하세요.)
  • 프롬프트 입력란에 이미지를 직접끌어다 놓 기
  • 이미지의파일 경로 제공

이 기능은 UI 개발의 기준점으로 디자인 모형을 사용하거나 분석 및 디버깅을 위한 시각적 차트로 작업할 때 특히 유용합니다. 컨텍스트에 시각적 요소를 추가하지 않는 경우에도 결과물이 시각적으로 매력적으로 보이는 것이 얼마나 중요한지 Claude에게 명확하게 설명하는 것이 도움이 될 수 있습니다.

c. Claude가 보거나 작업하기를 원하는 파일 언급하기

탭 완성 기능을 사용하여 리포지토리의 어느 곳에서나 파일이나 폴더를 빠르게 참조하여 Claude가 올바른 리소스를 찾거나 업데이트할 수 있도록 하세요.

d. Claude에게 URL 제공

Claude가 가져와서 읽을 수 있도록 특정 URL을 프롬프트와 함께 붙여넣으세요. 동일한 도메인(예: docs.foo.com)에 대한 권한 프롬프트를 피하려면 /permissions 을 사용하여 도메인을 허용 목록에 추가하세요.

e. 일찍 그리고 자주 코스 수정

자동 허용 모드(shift+탭을 눌러 전환)를 사용하면 Claude가 자율적으로 작업할 수 있지만, 일반적으로 적극적인 공동 작업자가 되어 Claude의 접근 방식을 안내하면 더 나은 결과를 얻을 수 있습니다. 처음에 클로드에게 작업에 대해 충분히 설명하면 최상의 결과를 얻을 수 있지만 언제든지 클로드의 코스를 수정할 수도 있습니다.

다음 네 가지 도구가 코스 수정에 도움이 됩니다:

  • Claude에게 코딩하기 전에계획을 세우도록 요청하세요. 계획이 제대로 된지 확인할 때까지 코딩하지 말라고 명시적으로 알려주세요.
  • 어떤 단계(생각하기, 도구 호출, 파일 편집)에서든 Claude를중단하려면 Escape 키를 눌러 컨텍스트를 유지하면서 지침을 리디렉션하거나 확장할 수 있습니다.
  • 기록을 되돌아가거나 이전 프롬프트를 편집하고 다른 방향으로탐색하려면 Escape 키를 두 번 누릅니다. 원하는 결과를 얻을 때까지 프롬프트를 편집하고 반복할 수 있습니다.
  • 다른 접근 방식을 취하기 위해 종종 2번 옵션과 함께변경 사항을 실행 취소하도록 Claude에게 요청합니다.

클로드 코드가 첫 번째 시도에서 문제를 완벽하게 해결하는 경우도 있지만, 일반적으로 이러한 수정 도구를 사용하면 더 나은 솔루션을 더 빨리 얻을 수 있습니다.

f. 컨텍스트에 집중하기 위해 /clear 사용

긴 세션 중에는 Claude의 컨텍스트 창이 관련 없는 대화, 파일 내용 및 명령으로 채워질 수 있습니다. 이로 인해 성능이 저하되고 때때로 Claude의 집중력이 흐트러질 수 있습니다. 작업 사이에 /clear 명령을 자주 사용하여 컨텍스트 창을 재설정하세요.

g. 복잡한 워크플로우에는 체크리스트와 스크래치패드 사용

코드 마이그레이션, 수많은 린트 오류 수정, 복잡한 빌드 스크립트 실행 등 여러 단계가 있거나 철저한 솔루션이 필요한 대규모 작업의 경우 Claude가 마크다운 파일(또는 GitHub 이슈!)을 체크리스트 및 작업 스크래치 패드로 사용하여 성능을 개선할 수 있도록 하세요:

예를 들어 많은 수의 린트 문제를 해결하려면 다음과 같이 할 수 있습니다:

  1. Claude에게 lint 명령을 실행하고 그 결과로 발생하는 모든 오류(파일 이름 및 줄 번호 포함)를 Markdown 체크리스트에 기록하라고지시합니다.
  2. Claude에게 각 문제를 하나씩 해결하고 수정 및 확인한 후 체크하고 다음 단계로 넘어가도록지시합니다.

h. Claude에게 데이터 전달

Claude에게 데이터를 제공하는 방법에는 여러 가지가 있습니다:

  • 프롬프트에 직접복사하여 붙여넣기 (가장 일반적인 방법)
  • Claude 코드 (예: cat foo.txt | claude)로 파이프 (특히 로그, CSV 및 대용량 데이터에 유용함)
  • bash 명령, MCP 도구 또는 사용자 지정 슬래시 명령을 통해데이터를 가져오도록 Claude에게 지시합니다.
  • Claude에게 파일을 읽거나 URL을 가져오도록요청 (이미지에도 작동)

대부분의 세션에서는 이러한 접근 방식을 조합하여 사용합니다. 예를 들어 로그 파일을 파이프로 가져온 다음 Claude에게 도구를 사용하여 추가 컨텍스트를 가져와 로그를 디버깅하라고 지시할 수 있습니다.

5. 헤드리스 모드를 사용하여 인프라 자동화하기

Claude Code에는 CI, 사전 커밋 후크, 빌드 스크립트 및 자동화와 같은 비대화형 컨텍스트를 위한 헤드리스 모드가 포함되어 있습니다. 헤드리스 모드를 사용하려면 -p 플래그와 함께 메시지를 표시하고, JSON 출력을 스트리밍하려면 --output-format stream-json 을 사용하세요.

헤드리스 모드는 세션 간에 지속되지 않는다는 점에 유의하세요. 매 세션마다 트리거해야 합니다.

a. 이슈 분류에 Claude 사용

헤드리스 모드는 리포지토리에 새 이슈가 만들어질 때와 같이 GitHub 이벤트에 의해 트리거되는 자동화를 강화할 수 있습니다. 예를 들어, 공개 Claude 코드 리포지토리는 새 이슈가 들어올 때 Claude를 사용하여 검사하고 적절한 레이블을 할당합니다.

b. Claude를 린터로 사용

Claude Code는 기존의 린팅 도구가 감지하는 것 이상의 주관적인 코드 리뷰를 제공하여 오타, 오래된 주석, 오해의 소지가 있는 함수 또는 변수 이름 등의 문제를 식별할 수 있습니다.

6. 멀티-Claude 워크플로우로 수준 향상

독립형 사용 외에도 가장 강력한 애플리케이션 중 일부는 여러 개의 Claude 인스턴스를 병렬로 실행해야 합니다:

a. 한 클라우드로 코드를 작성하고 다른 클라우드로 검증하기

간단하지만 효과적인 접근 방식은 한 명의 Claude가 코드를 작성하는 동안 다른 Claude가 이를 검토하거나 테스트하는 것입니다. 여러 엔지니어와 함께 작업할 때와 마찬가지로, 때로는 별도의 컨텍스트를 갖는 것이 유용할 수 있습니다:

  1. Claude를 사용하여 코드 작성하기
  2. /clear 을 실행하거나 다른 터미널에서 두 번째 Claude를 시작합니다.
  3. 두 번째 클로드가 첫 번째 클로드의 작업을 검토하도록 합니다.
  4. 다른 클로드(또는 /clear )를 다시 시작하여 코드를 읽고 피드백을 검토합니다.
  5. 이 클로드가 피드백을 바탕으로 코드를 수정하도록 합니다.

한 명의 클로드에게 테스트를 작성하게 한 다음 다른 클로드에게 테스트를 통과하도록 코드를 작성하게 하는 등 테스트에서도 비슷한 작업을 수행할 수 있습니다. 별도의 작업용 스크래치 패드를 주고 어느 쪽에 쓸지, 어느 쪽에서 읽을지를 알려주어 Claude 인스턴스가 서로 통신하도록 할 수도 있습니다.

이렇게 분리하면 하나의 Claude가 모든 것을 처리하는 것보다 더 나은 결과를 얻을 수 있습니다.

b. 리포지토리를 여러 번 체크아웃하기

Claude가 각 단계를 완료할 때까지 기다리는 대신, Anthropic의 많은 엔지니어는 이렇게 합니다:

  1. 별도의 폴더에3-4개의 git 체크아웃을 만듭니다.
  2. 별도의 터미널 탭에서각 폴더를 엽니다.
  3. 각 폴더에서 다른 작업으로Claude를 시작합니다 .
  4. 진행 상황을 확인하고 권한 요청을 승인/거부하기 위해순환합니다.

c. git 워크트리 사용

이 접근 방식은 여러 개의 독립적인 작업에 적합하며, 여러 개의 체크아웃에 대한 가벼운 대안을 제공합니다. Git 워크트리를 사용하면 동일한 리포지토리에서 여러 브랜치를 별도의 디렉터리로 체크아웃할 수 있습니다. 각 워크트리에는 격리된 파일이 있는 자체 작업 디렉터리가 있는 동시에 동일한 Git 기록과 리플로그를 공유할 수 있습니다.

git 워크트리를 사용하면 프로젝트의 여러 부분에서 각각 독립적인 작업에 초점을 맞춘 여러 개의 Claude 세션을 동시에 실행할 수 있습니다. 예를 들어, 한 클라우드는 인증 시스템을 리팩토링하고 다른 클라우드는 전혀 관련이 없는 데이터 시각화 컴포넌트를 빌드할 수 있습니다. 작업이 겹치지 않으므로 각 Claude는 서로의 변경 사항을 기다리거나 병합 충돌을 처리할 필요 없이 전속력으로 작업할 수 있습니다:

  1. 워크트리를 만듭니다: git worktree add ../project-feature-a feature-a
  2. 각 워크트리에서 Claude를 실행합니다: cd ../project-feature-a && claude
  3. 필요에 따라추가 워크트리를 생성합니다 (새 터미널 탭에서 1-2단계 반복).

몇 가지 팁

  • 일관된 명명 규칙 사용
  • 워크트리당 하나의 터미널 탭 유지
  • Mac에서 iTerm2를 사용하는 경우, Claude에게 주의가 필요할 때 알림을 설정하세요.
  • 다른 워크트리에 대해 별도의 IDE 창 사용
  • 작업이 끝나면 정리하세요: git worktree remove ../project-feature-a

d. 사용자 지정 하네스와 함께 헤드리스 모드 사용

claude -p (헤드리스 모드)는 내장된 도구와 시스템 프롬프트를 활용하면서 클로드 코드를 대규모 워크플로우에 프로그래밍 방식으로 통합합니다. 헤드리스 모드 사용에는 두 가지 기본 패턴이 있습니다:

1. 대규모 마이그레이션 또는 분석(예: 수백 개의 로그에서 감정 분석 또는 수천 개의 CSV 분석)을 처리하는 패닝 아웃:

  1. Claude에게 작업 목록을 생성하는 스크립트를 작성하게 하세요. 예를 들어, 프레임워크 A에서 프레임워크 B로 마이그레이션해야 하는 2k 파일 목록을 생성합니다.
  2. 각 작업에 대해 프로그래밍 방식으로 Claude를 호출하고 사용할 수 있는 작업과 도구 세트를 제공하면서 작업을 반복합니다. 예를 들어 claude -p “migrate foo.py from React to Vue. When you are done, you MUST return the string OK if you succeeded, or FAIL if the task failed.” --allowedTools Edit Bash(git commit:*)
  3. 스크립트를 여러 번 실행하고 프롬프트를 세분화하여 원하는 결과를 얻습니다.

2. 파이프라이닝은 기존 데이터/처리 파이프라인에 Claude를 통합합니다:

  1. claude -p “<your prompt>” --json | your_command, 여기서 your_command 은 처리 파이프라인의 다음 단계입니다.
  2. 여기까지입니다! JSON 출력(선택 사항)은 보다 쉬운 자동 처리를 위한 구조를 제공하는 데 도움이 될 수 있습니다.

이 두 가지 사용 사례 모두에서 Claude 호출 디버깅을 위해 --verbose 플래그를 사용하는 것이 도움이 될 수 있습니다. 일반적으로 프로덕션에서는 더 깔끔한 출력을 위해 자세한 정보 모드를 끄는 것이 좋습니다.
